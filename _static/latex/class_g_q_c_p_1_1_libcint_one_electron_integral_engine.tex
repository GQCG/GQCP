\subsubsection*{template$<$typename \+\_\+\+Shell, size\+\_\+t \+\_\+N, typename \+\_\+\+Integral\+Scalar$>$\newline
class G\+Q\+C\+P\+::\+Libcint\+One\+Electron\+Integral\+Engine$<$ \+\_\+\+Shell, \+\_\+\+N, \+\_\+\+Integral\+Scalar $>$}

An one-\/electron integral engine that uses libcint as its backend


\begin{DoxyTemplParams}{Template Parameters}
{\em \+\_\+\+Shell} & the type of shell the integral engine is able to handle \\
\hline
{\em \+\_\+N} & the number of components the operator has \\
\hline
{\em \+\_\+\+Integral\+Scalar} & the scalar representation of an integral\\
\hline
\end{DoxyTemplParams}
\begin{DoxyNote}{Note}
\+\_\+\+Shell is a template parameter because that enables compile-\/time checking of correct arguments.

Libcint1e\+Function and Libcint2e\+Function need the full libcint data (that is stored in a Raw\+Container), because the electron electrical dipole function needs to access the \textquotesingle{}common origin\textquotesingle{} and the nuclear attraction function needs to know all the nuclei that should be taken into account. We therefore keep the full Raw\+Container as a member.

In the overridden \mbox{\hyperlink{class_g_q_c_p_1_1_libcint_one_electron_integral_engine_adc39ac2900ad0612daf7ca0a57959e40}{calculate(const G\+T\+O\+Shell\&, const G\+T\+O\+Shell\&)}} method, the shell indices that correspond to the Raw\+Container should also be known, which is why we also keep the \mbox{\hyperlink{class_g_q_c_p_1_1_shell_set}{Shell\+Set}} as a member, for easy retrieval of the shell indices. 
\end{DoxyNote}
